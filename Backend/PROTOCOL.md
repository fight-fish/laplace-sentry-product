
# Laplace Sentry Backend — Protocol & Invariants

## 1. 文件定位

本文件為 Laplace Sentry Backend 的最高規範文件。

用於定義：

- 後端責任範圍
- 指令控制契約
- 核心資料流規範
- 模組職責邊界
- 系統不變量
- 前後端交互邊界
- 安全寫入與哨兵行為規則

當其他文件與本文件衝突時，以本文件為準。

---

## 2. 後端責任範圍（Scope）

Backend 為系統正式執行核心，負責：

- 專案設定管理
- 目錄結構解析
- 目錄樹生成
- Markdown 輸出格式處理
- 目標檔案安全寫入
- 背景哨兵監控
- 忽略與靜默規則管理
- CLI 指令處理
- Dispatcher 任務分派

Backend 不負責：

- 圖形介面
- 視覺呈現
- 使用者互動流程
- UI 字串管理

---

## 3. 系統不變量（Invariants）

### 3.1 正式入口不變量
所有正式操作必須透過：

main.py → daemon.py → 指定工作流程

不得繞過正式入口直接呼叫底層模組。

---

### 3.2 專案設定唯一來源
專案設定唯一真相來源為：

Backend/data/projects.json

禁止建立平行設定來源。

---

### 3.3 查詢與寫入分離
查詢行為僅生成目錄結構資料，不得修改目標檔案。  
正式寫入行為必須透過完整更新流程執行。

---

### 3.4 哨兵寫入限制
背景哨兵不得直接寫入最終輸出檔。  
正式寫入必須透過標準更新流程完成。

---

### 3.5 安全寫入不可繞過
所有檔案寫入必須經由安全寫入機制處理。  
不得直接覆蓋原始檔案。

---

## 4. CLI 控制契約（Command Protocol）

### 4.1 指令入口
所有 CLI 操作透過：

python main.py <command> [arguments]

---

### 4.2 指令處理流程
指令處理流程必須為：

main.py  
→ daemon.py  
→ dispatcher  
→ 對應處理模組

main.py 不承載業務邏輯。

---

### 4.3 操作模式

Backend 支援：

1. 互動式模式（不帶參數啟動）
2. 指令模式（帶參數單次執行）

指令模式供 UI 與腳本調用。

---

### 4.4 CLI 行為原則
CLI 操作必須：

- 回傳明確成功或失敗狀態
- 輸出格式穩定
- 避免依賴互動式輸入
- 可被自動化工具呼叫

---

## 5. 核心模組職責（Module Responsibilities）

### main.py
後端正式入口，負責模式判斷與指令轉發。

---

### daemon.py
後端主調度器，負責：

- 專案管理
- 哨兵管理
- 任務分派
- 更新流程觸發
- 查詢請求處理

---

### worker.py
單次更新流程協調器，串接資料生成與輸出流程。

---

### engine.py
目錄樹生成核心與資料結構處理。

---

### formatter.py
輸出格式處理，不負責資料生成。

---

### io_gateway.py
安全寫入處理，負責：

- 檔案鎖定
- 備份管理
- 原子寫入
- JSON 安全存取

---

### path.py
統一路徑處理模組，負責跨平台路徑正規化。

---

### sentry_worker.py
背景監控進程，負責目錄變化監控與更新請求觸發。

---

## 6. 資料契約（Data Contract）

### 6.1 projects.json
此檔為專案設定唯一來源，必須確保：

- 結構完整
- UUID 唯一
- 修改經由正式流程進行

---

### 6.2 合法修改方式
專案設定必須透過：

- CLI 指令
- Dispatcher 管理流程
- 安全寫入機制

禁止直接手動覆寫破壞結構。

---

## 7. 哨兵行為邊界（Sentry Boundaries）

背景哨兵允許：

- 監控目錄變化
- 節流高頻事件
- 更新靜默資訊
- 發送更新請求

背景哨兵禁止：

- 直接寫入 Markdown
- 修改專案設定
- 干涉 UI 狀態

---

## 8. 前後端邊界（Frontend–Backend Boundary）

Frontend 不得：

- 直接讀寫後端狀態檔
- 修改 projects.json
- 呼叫未公開的內部模組

Frontend 必須透過 CLI 或受控橋接層與 Backend 通訊。

---

## 9. 正式資料流（Formal Workflow）

正式更新流程：

main.py  
→ daemon.py  
→ worker.py  
→ engine.py  
→ formatter.py  
→ io_gateway.py

此流程為唯一正式資料流。

---

## 10. 重構安全原則（Refactor Safety）

未來重構時必須保持：

- CLI 契約穩定
- 專案設定來源不變
- 正式資料流不變
- 安全寫入機制存在
- 哨兵行為邊界不變

允許變動：

- 演算法優化
- 模組內部實作細節
- 效能優化
- 日誌細節調整
